- `auth` package (`backend/auth`): `login(input: LoginInput): Promise<LoginResult>` business logic in `src/login.ts`. Looks up the user by email, verifies the password with bcrypt (read-only, outside any transaction), then checks out a single client (`pool.connect()`) and runs the deactivate-prior-sessions (`UPDATE sessions SET is_active = false WHERE user_id = $1`) and insert-new-session as one `BEGIN`/`COMMIT` transaction (`ROLLBACK` + `release()` in a `finally` on error) â€” this closes a race where two concurrent logins for the same user could both deactivate-all then both insert, leaving two active sessions at once, which would defeat the single-active-session guarantee (`plan/15-account-security-anti-fraud.md`). Signs and returns a session JWT. Throws `Error('INVALID_CREDENTIALS')` for both unknown email and wrong password. Integration-tested in `tests/login.test.ts` against Postgres â€” 5/5 passing (added a concurrent-logins test asserting exactly one active session survives `Promise.all([login(...), login(...)])`) against the local Postgres instance. Wired into Lambda handler in `src/handler.ts` (Task 6).
- DB schema: `migrations/2_sessions_one_active_per_user.js` adds a partial unique index (`UNIQUE (user_id) WHERE is_active`) on `sessions`, enforcing "at most one active session per user" at the Postgres level. This closes the one residual gap the `login.ts` transaction fix (above) doesn't cover: a user's *first-ever* login (zero existing session rows) has no existing row for two concurrent transactions to serialize against, so both could still insert an active session. With the index in place, Postgres itself rejects the second concurrent insert with a unique-violation, which `handler.ts`'s existing catch-all (`err.message !== 'INVALID_CREDENTIALS'` â†’ 500) already turns into a plain 500 with no application code changes â€” turning "silently two active sessions" into "one request fails loudly," independent of any application-level transaction logic.
- `auth` handler (`backend/auth/src/handler.ts`): validates `req.body` against a `zod` schema (`src/schema.ts`) before calling `login()` â€” missing/wrong-typed fields (e.g. a missing `deviceId`, NOT NULL in `sessions`) get a `400 {"error": "Invalid request body"}` instead of reaching `login()` and failing as an opaque 500. The Express `app` is now exported separately from the `serverless-http`-wrapped `handler`, so `tests/handler.test.ts` can drive it directly with `supertest` â€” 4/4 passing: 200 with a token on correct credentials, 401 on wrong password, 400 on missing `deviceId`, and 500 (with `login()` stubbed via `vi.mock` for just that one case) when `login()` throws something other than `INVALID_CREDENTIALS`. `login.test.ts` and `handler.test.ts` share the same real Postgres tables and each reseed/reset them in `beforeEach`, so `auth/vitest.config.ts` sets `fileParallelism: false` to keep the two files from racing each other's setup.
- `auth` feature: `POST /auth/login` â€” email+password, creates a session, deactivates prior sessions, returns a JWT.
- `authorizer` feature: API Gateway custom authorizer â€” verifies JWT + checks the session is still active; exposes userId/tenantId/role/sessionId to downstream functions via request context. Throws `Error('Unauthorized')` uniformly for both a bad/expired token and a deactivated session (no distinction leaked to the caller). Integration-tested in `backend/authorizer/tests/handler.test.ts` against Postgres â€” 4/4 passing against the local Postgres instance (including the null-tenantId/super-admin case).

Anyone building/deploying this template from a workspace-based checkout needs to pass `--build-in-source` (or set it in `samconfig.toml`) for the same reason.
