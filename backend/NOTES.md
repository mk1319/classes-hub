# Backend — Current State

- Monorepo scaffold created: npm workspaces root, shared tsconfig base.
- Local Postgres dev environment via `docker-compose.yml` (postgres:16, credentials in `.env.example`) and `migrations/` folder for `node-pg-migrate`. In this dev machine specifically, Docker isn't installed; a native Postgres server (already running locally) was used instead — the `classeshub` role/database were created directly (`createdb -O classeshub classeshub`) to match the same `DATABASE_URL` the docker-compose setup would produce, so no code/config changed. `npm run migrate` (node-pg-migrate) applies cleanly against it.
- DB schema: `tenants`, `users`, `sessions` tables via `migrations/1_init.js`.
- Shared package `@classes-hub/shared`: `getPool()` (Postgres client), `signSessionToken`/`verifySessionToken` (JWT with `SessionClaims`: userId, tenantId, role, sessionId). Both pin the algorithm explicitly to `HS256` (`sign`'s `algorithm` option, `verify`'s `algorithms` allowlist) so a token can't be forced through a different/weaker algorithm.
- `tsconfig.base.json` has no `rootDir`/`outDir` — they broke `tsc --noEmit` for every extending package (relative paths in a base config resolve against the base file's own location, not the extender's). Lambda builds go through SAM's esbuild bundling, which doesn't need them.
- `auth` package (`backend/auth`): `login(input: LoginInput): Promise<LoginResult>` business logic in `src/login.ts`. Looks up the user by email, verifies the password with bcrypt (read-only, outside any transaction), then checks out a single client (`pool.connect()`) and runs the deactivate-prior-sessions (`UPDATE sessions SET is_active = false WHERE user_id = $1`) and insert-new-session as one `BEGIN`/`COMMIT` transaction (`ROLLBACK` + `release()` in a `finally` on error) — this closes a race where two concurrent logins for the same user could both deactivate-all then both insert, leaving two active sessions at once, which would defeat the single-active-session guarantee (`plan/15-account-security-anti-fraud.md`). Signs and returns a session JWT. Throws `Error('INVALID_CREDENTIALS')` for both unknown email and wrong password. Integration-tested in `tests/login.test.ts` against Postgres — 5/5 passing (added a concurrent-logins test asserting exactly one active session survives `Promise.all([login(...), login(...)])`) against the local Postgres instance. Wired into Lambda handler in `src/handler.ts` (Task 6).
- DB schema: `migrations/2_sessions_one_active_per_user.js` adds a partial unique index (`UNIQUE (user_id) WHERE is_active`) on `sessions`, enforcing "at most one active session per user" at the Postgres level. This closes the one residual gap the `login.ts` transaction fix (above) doesn't cover: a user's *first-ever* login (zero existing session rows) has no existing row for two concurrent transactions to serialize against, so both could still insert an active session. With the index in place, Postgres itself rejects the second concurrent insert with a unique-violation, which `handler.ts`'s existing catch-all (`err.message !== 'INVALID_CREDENTIALS'` → 500) already turns into a plain 500 with no application code changes — turning "silently two active sessions" into "one request fails loudly," independent of any application-level transaction logic.
- `auth` handler (`backend/auth/src/handler.ts`): validates `req.body` against a `zod` schema (`src/schema.ts`) before calling `login()` — missing/wrong-typed fields (e.g. a missing `deviceId`, NOT NULL in `sessions`) get a `400 {"error": "Invalid request body"}` instead of reaching `login()` and failing as an opaque 500. The Express `app` is now exported separately from the `serverless-http`-wrapped `handler`, so `tests/handler.test.ts` can drive it directly with `supertest` — 4/4 passing: 200 with a token on correct credentials, 401 on wrong password, 400 on missing `deviceId`, and 500 (with `login()` stubbed via `vi.mock` for just that one case) when `login()` throws something other than `INVALID_CREDENTIALS`. `login.test.ts` and `handler.test.ts` share the same real Postgres tables and each reseed/reset them in `beforeEach`, so `auth/vitest.config.ts` sets `fileParallelism: false` to keep the two files from racing each other's setup.
- `auth` feature: `POST /auth/login` — email+password, creates a session, deactivates prior sessions, returns a JWT.
- `authorizer` feature: API Gateway custom authorizer — verifies JWT + checks the session is still active; exposes userId/tenantId/role/sessionId to downstream functions via request context. Throws `Error('Unauthorized')` uniformly for both a bad/expired token and a deactivated session (no distinction leaked to the caller). Integration-tested in `backend/authorizer/tests/handler.test.ts` against Postgres — 4/4 passing against the local Postgres instance (including the null-tenantId/super-admin case).
- SAM template (`template.yaml`) wires `authorizer` as the default API Gateway authorizer and `auth` function at `POST /auth/login` (unauthenticated). `samconfig.toml` sets local build/deploy defaults (cached+parallel build, `classes-hub-backend` stack, `resolve_s3`, `CAPABILITY_IAM`).
  - Two fixes were needed vs. the original plan text to make `sam build` succeed: (1) `Metadata.BuildMethod: esbuild` must be a sibling of `Type`/`Properties` on each `AWS::Serverless::Function` resource, not nested inside `Properties` — nesting it there makes SAM's CloudFormation validator reject it as an unknown property. (2) This repo is an npm-workspaces monorepo (`@classes-hub/shared` is a workspace package, not published), so plain `sam build` fails trying to `npm install @classes-hub/shared` from the registry (404) when it copies each function folder into an isolated scratch build dir. Building with `sam build --build-in-source` makes SAM build in the real `auth/`/`authorizer/` folders, where npm can see the workspace's hoisted `node_modules` (including the `@classes-hub/shared` symlink), and esbuild bundles it straight into a single-file `handler.js` per function (verified by inspecting the built artifacts — no `@classes-hub/shared` import remains unresolved, and no `node_modules/` dir is needed alongside the bundle). Anyone building/deploying this template from a workspace-based checkout needs to pass `--build-in-source` (or set it in `samconfig.toml`) for the same reason.
  - `sam build` (with `--build-in-source`) succeeds: `Build Succeeded`, artifacts in `.aws-sam/build/`.
  - **Unverified in this environment / deferred:** `sam local start-api` (needs Docker, not installed here) and actual AWS deployment via `sam deploy` (no AWS credentials/region configured here, and `sam validate --lint` flags `nodejs20.x` as deprecated for new function creation as of this environment's clock — worth bumping the `Globals.Function.Runtime` before a real deploy). This exercises the same handler code that `sam local start-api`/AWS would run, just bypassing the API-Gateway-emulation layer itself.
